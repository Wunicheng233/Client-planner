# 智能移动技术大作业 - 仿真路径规划与控制

本项目是我们小组用于完成《智能移动技术》大作业的仿真代码仓库。通过与本地 Client 仿真环境进行 UDP 通信，我们将在此实现机器人的**全局路径规划**与**局部避障控制**。

## 项目结构说明

为了方便大家协作，目前项目代码已经完成了高度的面向对象封装。核心文件结构如下：

*   **`main.py`**：主程序入口，仅负责模块的实例化和主循环启动。**请尽量不要修改此文件**，保持其简洁。
*   **`utils.py`**：**核心主战场**。包含核心控制逻辑封装 `RobotController`（状态机管理、防撞检测、PD 局部控制、绘图等）。我们后续的优化工作主要集中在这里。
*   **`a_star_planner.py`**：全局路径规划器。目前实现了基础的 A* 算法和简单的障碍物获取逻辑，可以在此拓展更高级的算法。
*   **`vision.py` / `action.py` / `debug.py`**：底层 UDP 通信与接口封装（读取视觉、发送速度、仿真器绘图），保持原样即可。
*   **`*_pb2.py`**：Protobuf 自动生成的数据结构文件，**绝对不要修改**。

## 团队目标与评分标准

根据课程要求，我们需要在仿真环境中控制小车（蓝车 0 号）完成 **5 个来回**的往返（起点 `(2500, 1800)` ↔ 终点 `(-2400, -1500)`）。最高物理限速：线速度 5000 mm/s，角速度 10 rad/s。

期末验收满分 8 分，具体得分点如下，这也是我们后续优化的**核心方向**：

1.  **基础跑通 (5分)**：程序不死锁、不 Crash，能跑完 5 个来回。（已完成）
2.  **SIMPLE 模式 - 静态环境 (2分)**：
    *   **时间表现 (0.8分)**：总耗时 **< 100秒 (0.8分)**；100~150秒 (0.6分)；150~200秒 (0.4分)。（已完成）
    *   **碰撞控制 (0.6分)**：每次碰撞扣 0.1 分，力争 **0 碰撞**。（**重点优化方向**）
    *   **轨迹平滑性 (0.6分)**：轨迹整体平滑、无明显原地打转、无频繁急停急转得满分。（**优化方向**）
3.  **HARD 模式 - 动态环境 (1分)**：
    *   **通过能力 (0.4分)**：能在动态黄车干扰下稳定完成任务得满分。
    *   **时间表现 (0.3分)**：总耗时 **< 100秒 (0.3分)**；100~150秒 (0.2分)。（已完成）
    *   **碰撞控制 (0.3分)**：碰撞 **< 4次 (0.3分)**；4~6次 (0.1分)。（重点优化方向）

## 环境配置

为了避免依赖冲突，团队统一使用 Conda 管理开发环境。请严格按照以下步骤进行配置：

**1. 克隆代码仓库**
在终端中执行以下命令将代码拉取到本地：
```bash
git clone [https://github.com/Wunicheng233/Client-planner.git](https://github.com/Wunicheng233/Client-planner.git)
cd Client-planner
```

**2. 创建并激活 Conda 环境**
我们的程序要求 Python 版本 >= 3。推荐使用 Python 3.8：
```bash
conda create -n move python=3.8 -y
conda activate move
```

**3. 安装核心依赖**
本项目依赖 `protobuf` 进行数据序列化。**注意：版本过高会导致运行时直接报错退出。** 我们就统一指定安装 `3.20.0` 版本：
```bash
pip install protobuf==3.20.0
```

## 运行与测试

在确认环境配置无误后，可以通过以下步骤跑通基础 Demo：

1.  **启动仿真器**：打开桌面的 `Client` 软件。
2.  **连接图像**：点击右侧的“连接图像”按钮（带有插头图标），并选择测试场景（初期测试推荐选择 `SIMPLE` 静态场景）。
3.  **运行主程序**：在激活了 `move` 环境的终端中，运行以下命令：
    ```bash
    python main.py
    ```

> **成功标志**：如果终端成功打印出 `>>> A* + PD 追踪器就绪 <<<` 等提示，且 Client 中的蓝车 0 号开始向目标点移动，并在界面上画出红色规划路径与目标圆圈，说明环境已配置完毕！
>
> *(ps：在 Client 软件中，按住 `Ctrl` 键加上数字键可以快速控制场上动态黄车的数量，鼠标也可直接拖拽小车)*

## 协作开发指南与优化方向

目前的基线版本（最保守的 A* + PD 控制）能够跑通流程拿及格分，但在速度和动态避障上还有极大的优化空间。为了拿到高分，建议大家从以下几个方向认领任务（主要修改 `utils.py`）：

1.  **局部避障与防撞逻辑重构 (`check_path_blocked`)**：目前的防撞极其简单粗暴（距离障碍物 < 500mm 就直接清空全局路径要求重算 A*），这在 HARD 模式下面对乱跑的黄车很容易卡死。**优化建议**：引入 **DWA（动态窗口法）** 或 **APF（人工势场法）** 进行局部平滑绕行。
2.  **速度规划与平滑控制 (`calculate_pd_control`)**：目前的 PD 控制器在转弯时会狠狠降速，且为了求稳最大线速度只敢给到 2500。**优化建议**：调优 PD 参数（或升级为 PID / Pure Pursuit 纯跟踪算法），让小车在直道敢于加速到满速 5000，弯道过渡更加丝滑，把总耗时压进 100 秒内。
3.  **动态障碍物预测 (`get_obstacles`)**：目前的障碍物获取是“眼见为实”（只取当前时刻坐标）。**优化建议**：利用 `vision` 模块读取动态黄车的速度矢量 (`vel_x`, `vel_y`)，预测其未来 1-2 秒的轨迹位置，并在代价地图中提前膨胀避让。
4.  **路径平滑 (`smooth_path`)**：优化现有的多次线性插值平滑算法，可以考虑引入 B 样条曲线 (B-Spline) 或贝塞尔曲线，提升平滑度得分。

> **Git 协作规范**：
> 每个人开发新功能或调参时，请务必**新建分支**（如 `git checkout -b opt-dwa`）。在本地 `SIMPLE` 和 `HARD` 模式下跑满 5 个来回且确认成绩提升后，再发起 Merge Request 合并到 `main` 分支。

## 常见错误排查 (Troubleshooting)

| 报错现象 / 提示语 | 产生原因 | 解决办法 |
| :--- | :--- | :--- |
| **Client 运行白屏** | OpenGL 渲染库缺失 | 在系统 `C:\Windows\System32` 路径下找到 `opengl32.dll`，将其复制并覆盖粘贴到 Client 的安装目录下。 |
| **`TypeError: Descriptors cannot not be created directly`** | `protobuf` 版本过高 | 在终端中执行 `pip install protobuf==3.20.0` 进行降级。 |
| **`OSError: [WinError 10048] 通常每个套接字地址只允许使用一次`** | UDP 端口被占用 | 请确保后台没有运行多个 Python 实例（注意检查 IDE 终端里没关掉的隐藏进程）。在 Client 软件中点击“快速关闭程序”（靶心图标）彻底关闭软件后，再重新打开重试。 |
| **终端无报错，但小车不动 / 视觉坐标全是 -999999** | 视觉模块未成功建立连接 | 主程序初始化已加入 `time.sleep(0.1)` 等待，若仍出现，请尝试在 Client 中断开图像连接后再重新连接。 |
